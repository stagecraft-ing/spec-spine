# 020: Derived-artifact merge driver and merge-queue serialization

## 1. Purpose

spec-spine commits its two deterministic derived artifacts so the coupling gate
(spec 005) and the content-hash staleness check (spec 004) can compare the
committed ledger against current inputs:

- `.derived/spec-registry/registry.json` (the compiler output), and
- `.derived/codebase-index/index.json` (the indexer output).

Both embed a global `contentHash`. That is exactly what makes them mergeable in
principle (determinism guarantees the same inputs produce the same bytes), but a
plain textual merge of two branches that each regenerated an artifact conflicts
on the hash line and the changed records. Resolving that by hand, or by a
regenerate-and-recommit dance on every rebase, is friction this spec removes.

This is a port of the proven OAP `oap-index-regen` mechanism (OAP spec 188),
adapted to spec-spine's two-artifact reality and rebranded.

## 2. Territory

- **`.githooks/`** (directory subtree): the merge-driver implementation
  (`merge-derived-index.sh`) and its one-command per-clone registrar
  (`enable-merge-driver.sh`).

Two supporting surfaces are wired but deliberately not claimed as units, because
they sit on the coupling bypass floor (`.github/`) or are unowned root config,
and over-claiming them would create coupling friction for routine edits:

- **`.gitattributes`**: carries the committed path to driver assignment
  (`merge=spec-spine-derived-regen`) for both artifacts. The assignment travels
  with the repo; the registration that activates it does not (see below).
- **`.github/workflows/ci.yml`** and **`.github/workflows/determinism.yml`**:
  gain the `merge_group` trigger.

## 3. Behavior

### 3.1 The merge driver

`.githooks/merge-derived-index.sh` is a git low-level merge driver invoked as
`<driver> %O %A %B %P`. It MUST:

- locate a usable `spec-spine` binary (a release or debug build under `target/`,
  else one on `PATH`), and if none is found, **fail closed**: exit non-zero and
  leave the conflict in place for manual resolution;
- regenerate **both** artifacts from the merged working tree by running
  `spec-spine compile` then `spec-spine index`; if either fails, fail closed;
- copy the freshly regenerated artifact named by `%P` over the `%A` (ours) temp
  file and exit zero, so git records the regenerated bytes as the resolution.

A single driver script handles both paths: git invokes it once per conflicting
path with the path in `%P`, and regenerating both artifacts is idempotent.

**Known limitation (ported from OAP spec 188):** regeneration is only as
complete as the working tree at git's driver-invocation moment, so on some
rebases the auto-heal can emit a clean-but-stale hash. The CI staleness gate
(`spec-spine index check`, spec 004) is the freshness source of truth, not this
driver. The driver never weakens that gate; it only avoids a textual conflict.

### 3.2 Registration (opt-in, per clone)

The path to driver assignment lives in committed `.gitattributes`:

```
.derived/spec-registry/registry.json   merge=spec-spine-derived-regen
.derived/codebase-index/index.json      merge=spec-spine-derived-regen
```

The driver registration lives in per-clone `.git/config`, which is not
committed, so each clone activates it once with `.githooks/enable-merge-driver.sh`.
Absent that registration git falls back to the default text merge, so the
feature is strictly opt-in and inert by default: no behavior change for a clone
that has not enabled it.

### 3.3 Merge queue and the `ci-gate` check

`ci.yml` gains a `merge_group` trigger so every PR in the GitHub merge queue is
validated against the speculative merged tree, catching pairs of PRs that pass
alone but break together (for example a duplicate spec id introduced on two
branches). The cross-triple determinism golden runs as a reusable workflow
(`determinism.yml`, now `workflow_call`) invoked by `ci.yml`, so the queue covers
it too. The build/test/clippy/fmt job, the self-governance
`compile`/`index check`/`lint` steps, and the determinism gate all run on
`merge_group`.

`ci.yml` ends in a single **`ci-gate`** aggregator job that `needs` every other
CI job (test, dogfood, determinism) and fails if any of them failed or was
cancelled (a skipped job counts as a pass). Branch protection and the merge
queue require just this one check name, rather than an enumerated, drift-prone
list of individual job names.

The coupling gate (spec 005) MUST remain `pull_request`-only. It reads the
`Spec-Drift-Waiver:` line from the PR body, which the `merge_group` event does
not carry; running it in the queue would have no waiver source. Every PR has
already passed the coupling gate before it enters the queue, so omitting it from
`merge_group` loses no coverage.

The queue is **inert until a repository admin enables it** in branch protection
and requires the `ci-gate` check. Adding the trigger and the gate job is safe
and has no effect until that operational step.

## 4. Out of scope

- Auto-resolving conflicts in any file other than the two committed derived
  artifacts. Everything else uses git's normal merge.
- Replacing or relaxing the staleness gate (spec 004) or the coupling gate
  (spec 005). The driver is purely a conflict-resolution convenience.
- Enabling the merge queue in repository settings. That is an operational step a
  human takes once; this spec only ships the inert workflow triggers.
